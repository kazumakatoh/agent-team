# 健康データ統合・分析システム 設計書（v0.1 ドラフト）

> **このドキュメントの位置付け**
> 加藤社長との初回ヒアリング（2026-04-27）に基づく完成イメージのドラフト。
> 社長レビュー → 修正 → 確定 → 構築開始 の流れ。

---

## 1. 目的

健康診断・各種診断データ・日々のバイタルを蓄積し、

- **点ではなく線・面で見る**：単発の数値ではなく、年次推移・項目間の相関で俯瞰
- **正しく認識する**：基準値・年齢相応・自分自身の経時変化の3軸で評価
- **健康寿命を延ばす**：早期発見＋実行可能な改善アクション提案

を実現する仕組みを構築する。

## 2. ヒアリング結果サマリー

| 項目 | 決定内容 |
|---|---|

## 3. 完成イメージ（運用ストーリー）

### A. 健診結果が出たとき（年1〜2回）
1. 社長が健診結果PDFを Drive `01_inbox/` に保存
2. GASが新規PDFを検知し、Claude APIで項目抽出
3. 項目辞書で正規化されたデータが `master_health` に追記
4. Claudeが**健診サマリーレポート**を `analysis/` に出力
   - 前年比較
   - 5年推移グラフ（基準値逸脱項目を赤表示）
   - 生活習慣病リスクスコア（簡易）
   - 次回健診までの**改善アクションプラン＋KPI**
5. 社長が確認、誤抽出は手動修正

### B. 月次バイタルレビュー（毎月初）
1. GASがGoogle Fit APIから前月のバイタルデータをプル
2. `master_vitals` に書き込み
3. Claudeが**月次バイタルレビュー**を生成
   - 月平均（歩数・睡眠・心拍・体重・体脂肪）
   - 前月比、3ヶ月トレンド
   - 直近健診の懸念項目との関連推測（例：体重↑＋歩数↓ → 脂質悪化リスク）
   - 当月の重点アクション提案
4. 社長が朝のレポート時に確認

### C. 四半期レビュー（年4回）
1. Claudeが**直近健診×3ヶ月バイタル横断分析**を生成
2. 改善アクションプランの達成度評価
3. 次四半期のKPI更新提案

### D. アラート（随時）
- 基準値の重大な逸脱、急激な変化があれば、次のレビューを待たずに秘書エージェント経由で通知

## 4. システム構成図

```
[社長]
   │ PDFを置く / レポートを読む
   ▼
[Google Drive：制限フォルダ]
   ├ 01_inbox/                  ← 健診PDFを置くだけ
   ├ 02_processed/              ← 処理済みPDFアーカイブ
   ├ master_health.gsheet       ← 健診マスター（全項目時系列）
   ├ master_vitals.gsheet       ← バイタルマスター（日次）
   ├ item_dictionary.gsheet     ← 項目名正規化辞書
   └ analysis/                  ← AIレポート出力先
        ├ 2026_annual.md
        ├ 2026Q2_quarterly.md
        └ 2026-04_monthly_vitals.md
   ▲
   │
[Google Apps Script]
   ├ inbox_watcher              ← 新規PDF検知（時限トリガー）
   ├ pdf_extractor              ← Claude APIにPDF送信→JSON取得
   ├ sheet_writer               ← master_healthへ追記、重複チェック
   ├ fitbit_pull                ← Google Fit API月次プル
   └ report_trigger             ← Claude APIにレビュー依頼
   ▲
   │
[Claude API]
   ├ PDF→構造化データ抽出
   ├ 健診サマリー生成
   ├ 月次バイタルレビュー
   ├ 四半期総合分析
   └ アラート判定
```

## 5. データモデル

### 5.1 健診マスター（`master_health`）
1行 = 1検査日 × 1項目（ロングフォーマット）

| 列名 | 例 | 備考 |
|---|---|---|
| exam_date | 2026-04-15 | 検査日 |
| facility | ○○クリニック | 受診医療機関 |
| category | 定期健診 | 定期健診 / 人間ドック / 個別検査 |
| item_code | hba1c | 辞書コード（正規化済み） |
| item_name_raw | HbA1c | PDF表記そのまま |
| value | 5.4 | 数値 |
| unit | % | 単位 |
| ref_low | 4.6 | 基準値下限 |
| ref_high | 6.2 | 基準値上限 |
| judgment | A | A/B/C/D 等 |
| source_pdf | 20260415_xxclinic.pdf | 出典PDFファイル名 |
| extracted_by | AI | AI / 手動 |
| confirmed | TRUE | 社長確認済みフラグ |
| note | | 任意メモ |

### 5.2 項目辞書（`item_dictionary`）
PDF間の表記揺れを統一コードに正規化。

| 列名 | 例 |
|---|---|
| item_code | hba1c |
| canonical_name | HbA1c（NGSP） |
| aliases | HbA1c, ヘモグロビンA1c, グリコヘモグロビン |
| category | 糖代謝 |
| unit_canonical | % |
| ref_low_default | 4.6 |
| ref_high_default | 6.2 |
| risk_focus | 生活習慣病 |
| description | 過去1〜2ヶ月の平均血糖値の指標 |

**初期辞書**：一般的な健診項目（30〜50項目程度：肝機能・脂質・血糖・腎機能・血液・血圧・身体計測など）を予め登録。

### 5.3 バイタルマスター（`master_vitals`）
1行 = 1日（ワイドフォーマット）

| 列名 | 例 |
|---|---|
| date | 2026-04-01 |
| steps | 8520 |
| resting_hr | 62 |
| sleep_hours | 6.8 |
| weight_kg | 72.3 |
| body_fat_pct | 18.5 |
| active_minutes | 35 |
| source | google_fit |

## 6. AI分析テンプレート

### 6.1 健診サマリー（年次）
- 受診情報（日付・機関・カテゴリ）
- **総合所見**（3行サマリー）
- 項目別ステータス（基準値内 / 軽度逸脱 / 要注意 / 要受診）
- 前年比・5年推移
- **生活習慣病リスク評価**（肝機能・脂質・血糖・血圧・肥満の5領域でスコア化）
- **改善アクションプラン**（次回健診までのKPI付き）

### 6.2 月次バイタルレビュー
- 月平均と前月比、3ヶ月移動平均
- 異常値日（睡眠不足5日連続など）
- 健診懸念項目との関連推測
- 当月の重点アクション

### 6.3 四半期総合分析
- 健診×バイタル横断
- 改善アクションプラン進捗（達成度％）
- 軌道修正提案
- 次四半期KPI

### 6.4 アラート
- トリガー：基準値の30%以上逸脱、3ヶ月連続悪化、急変
- 配信先：このフォルダ＋秘書エージェント経由

## 8. セキュリティ・プライバシー

- **保管先**：会社Drive `01_経営/01_社長個人/01_健康/`（社長アカウントのみ閲覧可、共有設定**個人のみ**）
- **外部共有**：禁止（医師相談用は手動エクスポートのみ）
- **AI送信時のマスキング**：氏名・住所・保険証番号・電話番号は抽出時に除去
- **バックアップ**：月次でDriveバックアップ（自動）
- **アーカイブ**：5年経過後のPDFは社長判断でアーカイブ
- **アクセスログ**：GAS実行ログを `analysis/_log/` に保管
