| 指標 | 使用する場面 | 計算例 |
|---|---|---|
| **CV（注文件数）** | 広告効果の評価 | CPA = 広告費/CV, CVR = CV/セッション |
| **注文点数（販売数）** | 販売実績・在庫・原価計算 | 仕入原価 = 仕入単価 x 点数, 在庫回転率 |

## 通知設計

### 週次改善提案（Gmail）

- **送信タイミング**: 毎週月曜 8:00
- **宛先**: 社長個人メールアドレスのみ
- **生成方法**: GAS → Claude API (claude-sonnet-4-6) → Gmail送信
- **履歴保管**: Gmail受信箱のみ

### 月次改善提案（Gmail）

- **送信タイミング**: 毎月1日 9:00
- **生成AIモデル**: Claude Opus 4.6 (claude-opus-4-6)
- **内容**: 月次サマリー + 全カテゴリ提案 + 前月比 + 戦略立案

### 緊急アラート（LINE）

- **送信方法**: LINE Messaging API
- **宛先**: 社長個人のみ
- **トリガー条件**:
  - 在庫切れ間近（残り在庫 < 直近7日平均売上 x 3日分）
  - 1日の広告費が異常値（直近7日平均の2倍超）
  - Settlement Report で大きな返金・調整金が発生
  - アカウント健全性の警告検出

## 改善提案の内容（5カテゴリ + 戦略立案）

### ① 商品登録チェック

| チェック項目 | データ取得方法 | 確認内容 |
|---|---|---|
| Amazon VINE | SP-API Catalog Items | 登録されているか |
| A+コンテンツ | SP-API Catalog Items | 設定されているか |
| 広告設定 | Ads API | M19でキャンペーンが作成されているか |
| 商品画像 | SP-API Catalog Items | メイン画像+サブ画像の数 |
| 商品タイトル・箇条書き | SP-API Catalog Items | 最適化されているか |

### ② 販売改善

| 分析内容 | データソース |
|---|---|
| KPI目標との比較 | kpi_targets.md の目標値 vs 実績 |
| レビューチェック | SP-API Reviews（マイナスレビューの内容要約） |
| CVR改善ポイント | トラフィックデータから訪問→CV の課題特定 |
| 価格適正性 | 現在価格 vs 粗利率の分析 |

### ③ 競合チェック

| 分析内容 | データソース |
|---|---|
| 競合価格比較 | SP-API Product Pricing API |
| 競合の広告強度 | Ads API 検索用語レポート（競合キーワードの出現頻度） |
| Buy Box獲得状況 | SP-API Traffic Report (Buy Box率) |
| 新規プロモーション施策提案 | Claude API による競合分析ベースの提案 |

### ④ セール対策

| 分析内容 | データソース |
|---|---|
| 年間セールカレンダー | 手動管理（プライムデー/BF/年末等） |
| セール前の対策ポイント | Claude API が過去データ + セール時期から提案 |
| 在庫準備の警告 | 在庫データ vs 過去セール時の販売増加率 |
| クーポン・値引き施策 | Claude API による最適施策の提案 |

### ⑤ アカウント健全性

| チェック項目 | データソース |
|---|---|
| アカウントヘルス警告 | SP-API Account Health |
| 在庫切れ商品 | Inventory API |
| 出品ステータス異常 | SP-API Catalog Items |
| 返品率異常 | 返品データの急増検知 |
| その他エラー | SP-API通知・アラート |

### + 戦略立案（AI逆算分析）

Claude API (Opus 4.6) が以下を月次で生成:
- 利益最大化のための逆算分析（目標利益→必要売上→必要CVR→必要セッション数）
- 広告予算の最適配分提案（ASIN別のROAS比較から予算シフト案）
- 撤退・注力の判断材料提示
- 次月のアクションプラン（優先度付き）

## GAS 自動実行スケジュール

| 実行時刻 | 処理内容 |
|---|---|
| 毎日 6:00 | SP-API + Ads API データ取得 → D1, D3 更新 |
| 毎日 6:30 | L1, L2, L3 ダッシュボード再集計 |
| 毎週月曜 8:00 | Claude API → Gmail で週次改善提案送信 |
| 毎月1日 8:00 | L1 月次サマリー更新 |
| 毎月1日 9:00 | Claude API (Opus) → Gmail で月次改善提案 + 戦略立案 |
| Settlement到着時 | D2 経費明細取り込み・D1の暫定→確定更新 |

### GAS実行時間制限への対応

- SP-APIレポート取得を分割（createReport / download を別トリガー）
- reportId を ScriptProperties に永続化
- 商品数が多い場合はバッチ処理（50商品ずつ等）

## 既存Excel資産の移行

### 移行対象

| 種別 | シート数 | 移行先 |
|---|---|---|
| Weekly シート | 10 | D1（日次データ）に統合 |
| Monthly シート | 8 | D1（日次データ）に統合 |
| Amazon全体集計 | 12 | L1（事業ダッシュボード）で再現 |
| 商品別タブ | 各商品分 | L3（商品分析）で動的表示に統合 |

### 移行方針

- 商品マスター（ASIN・仕入単価・仕入先）→ M1 + M2 へ移行（最重要）
- 過去の売上・広告実績 → D1 に一括インポート（過去3年分）
- 前年同月比・年次比較で活用
- 既存の経費内訳（Amazon割引/手数料/配送料等）→ D2 のカテゴリとマッピング
